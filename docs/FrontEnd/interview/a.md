## API 請求生命週期管理

因應快速切換畫面，設計API 請求生命週期管理機制，避免效能浪費

- 每次切換路由時，先把還沒完成的 API 請求全部取消
- 使用者快速從 A 頁切到 B 頁時，A 頁的 API 晚一點才回來，結果把 B 頁畫面狀態蓋掉
- 已經離開的頁面還繼續打 API，浪費頻寬和記憶體
- 重複點擊切頁造成一堆無效請求累積

1. beforeEach攔截
2. 發送API時，都會把請求家近pendingControllers.add(controller)
3. 那切換頁面時，就會把pendingControllers中還沒結束的Abort掉

## 路由資源拆分與預載策略

主導前端路由資源拆分與預載策略設計，降低首屏資源阻塞並優化使用者互動流暢度

- 路由主要是依照服務類別去區分，讓後續工程師在維護時，比較好依照服務去做修改
- 預載策略：主要是當有同級頁面時，會先去撈取同級的文件預先加載
- Lazy Import：不會一次把所有JS打包，要用到才去做打包
- 先做路由層級的 lazy loading，把各功能模組拆成獨立 chunk，避免首屏同步載入過多非必要資源；在此基礎上，再補一層預載策略，等當前頁面穩定後用 idle 時間預先載入高機率切換的兄弟頁，兼顧首屏速度和後續操作流暢度。

## Infinite Scroll 與 Debounce / Throttle

實作 Infinite Scroll 與 Debounce 機制

- 這主要就是去判斷說滾動是否到底了，如果到底就往下一頁這樣
- Debounce：防抖：這段時間內「沒有」再次觸發，才執行動作。如果這段時間內又觸發了，就重新計時。
- Throttle：節流：無論你觸發了多少次事件，動作只會執行一次。它保證了執行頻率。

建構 Web 操作自動化流程，降低人工操作成本

### 自動化工具開發Selenium：

- 使用 Selenium 建構瀏覽器自動化流程，處理登入、頁面操作、資料擷取與表單送出等情境，降低人工操作成本。
- 實作上會由 Python 接管既有瀏覽器 session，例如透過 `debugger_address=127.0.0.1:9222` 連到已啟動的 Chrome，方便沿用既有登入狀態與操作環境。
- 在擷取資料或執行操作前，會先確認元素已經出現在畫面且可互動，避免因頁面尚未完成渲染而造成腳本不穩定；這類等待通常會搭配 `WebDriverWait` 與 `visibility_of_element_located` 使用。

#### 常用操作與等待策略

- `webdriver.Chrome(options=options)`：建立瀏覽器控制實例
- `WebDriverWait(driver, timeout).until(...)`：等待指定條件成立後再繼續執行
- `visibility_of_element_located`：確認元素已可見再操作，提升流程穩定性
- `btn.click()`、`input.send_keys()`、`input.clear()`、`form.submit()`：模擬使用者操作
- `.text`、`.get_attribute()`：擷取畫面文字與元素屬性

#### 元素定位方式

- `By.CSS_SELECTOR`：實務上最常用，語法直觀、效能也較好
- `By.XPATH`：適合處理巢狀結構複雜、缺少明確 class 或 id 的情境
- `By.ID` / `By.NAME` / `By.CLASS_NAME`：在欄位規則明確時可直接使用

#### 瀏覽器設定與執行環境

- `--headless`：無頭模式，常用於伺服器或排程任務
- `--disable-gpu`：部分 Linux / headless 環境下會搭配使用
- `--window-size=1920,1080`：固定視窗尺寸，降低版面差異造成的定位問題
- `--incognito`：需要隔離瀏覽器狀態時可使用
- `--no-sandbox`、`--disable-dev-shm-usage`：常用於 Docker 或資源受限環境，降低瀏覽器崩潰機率

#### 自動化特徵處理

- `--disable-blink-features=AutomationControlled`：降低自動化標記被偵測的機率
- `excludeSwitches=["enable-automation"]`：移除部分自動化提示
- 自訂 `user-agent`：模擬一般使用者瀏覽器環境

#### 面試可怎麼描述

- 我會把 Selenium 的經驗描述成「用程式穩定接管瀏覽器操作流程」，重點不只是點擊或抓資料，而是如何處理等待機制、定位策略、登入狀態沿用與執行環境穩定性。
- 如果頁面結構單純，我會優先用 CSS Selector；如果是動態內容或結構複雜的區塊，才會改用 XPath。
- 自動化流程真正的難點通常不是 API 使用，而是讓腳本在不同環境、不同載入速度下仍然維持穩定與可重複執行。

## API 請求紀錄與追蹤

建立 API 請求紀錄與追蹤機制

1. 請求開始時會記錄時間，完成時會統一寫 log，內容包含 request method、host、url、status、duration。
2. 伺服器端會把紀錄寫進 logs/server-日期.log，也就是說不是只看 console，而是有落地的請求日誌。
3. 如果 response status 是特定錯誤，例如 403，也會統一觸發前端錯誤通知，讓使用者與開發都能看到一致的錯誤行為。

## 負責系統改版與 RWD 重構，提升整體 UI 一致性

- 主要參與整體網站改版，從舊版介面逐步重構為新的 UI / UX 規格，包含頁面視覺、元件行為與互動流程的全面調整。
- 在實作上不只是單純切版，而是把原本分散、樣式不一致的頁面，整理成可重用的版型與共用元件，讓後續新增功能時可以沿用同一套結構。
- RWD 重構的重點是重新定義桌機、平板、手機的斷點與版面規則，避免不同頁面各自處理，降低維護成本，也讓整體操作體驗更一致。
- 比較困難的點在於，要在不影響既有功能的前提下，逐步替換舊頁面，控制樣式回歸風險與跨裝置顯示問題。

## 開發與重構商業邏輯模組，提升程式可維護性

- 主要針對專案內重複出現的商業邏輯，避免相同邏輯散落在多個頁面。
- 重構時會先區分哪些是純函式邏輯、哪些和畫面或框架生命週期耦合，前者優先抽成共用函式庫，後者則整理成 composable 或 service 層，讓責任邊界更清楚。
- 這樣做的好處是後續需求變更時，不需要每個頁面都同步修改，可以集中調整，降低漏改與回歸風險。
- 從工程面來看，重構的核心不是把程式拆小而已，而是建立可預測的資料流與模組邊界，讓新功能能疊加在既有架構上，而不是持續堆疊例外邏輯。

## 設計並導入 i18n 多語系架構，支援跨產業應用

- 「把產品內容從程式碼中解耦」，讓前端架構更能支援商業擴張，而不只是做表面的多語切換。
- 因為產品本身有跨客戶、跨產業複用的需求，不只是語言切換而已，實際上還包含文案、品牌名稱、部分內容配置的差異，所以當初導入 i18n 架構來支撐這件事情。
- 我的做法是先把原本寫死在頁面中的文案抽離，統一收斂到語系資源檔，再建立對應的 key 命名規則，避免後續維護時找不到來源或重複定義。
- 這樣的好處是未來如果同一套產品要快速切成不同品牌版本，可以透過切換語系資源與配置內容完成大部分調整，不需要直接改動頁面邏輯。

- 如果在面試中談這段，我會強調自己除了開發功能，也會站在交付角度思考，確保版本可以穩定地從開發推進到實際上線。

## 效能優化與重構

- 重構既有程式架構，將系統等待時間縮短約 50%。
- 當時主要是針對單一檔案內過多且重複的迴圈處理做調整，包含像多層巢狀迴圈這類高成本邏輯，重新整理資料處理流程與計算方式。
- 重構後不只是程式碼可讀性提升，實際開窗或載入等待時間也明顯下降，整體效能大約改善一半。
- 這段經驗我會用來說明，效能優化不一定都來自大型架構改動，很多時候是先找到熱點邏輯，針對資料處理流程做有依據的重構，就能帶來很明顯的改善。

## 前後端 API 協作

- 與後端（Java）協作定義 API 規格，並負責前端畫面整合與資料呈現。
- 在串接過程中，我會針對欄位命名、資料結構、狀態碼與錯誤情境先和後端對齊，避免前端做到一半才發現規格不完整。
