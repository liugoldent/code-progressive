# [FE] System - 2 千萬級高並發秒殺架構設計

## 這篇在講什麼

- 這篇文章在講的是「秒殺系統」怎麼扛住極短時間內暴增的流量。
- 核心不是只做快，而是要同時解決高併發、不能超賣、不能少賣、不能把資料庫打爆這幾個問題。
- 文章用 12306 抢票的場景來說明，整體重點是架構分層、流量削峰、預扣庫存、異步下單與超時釋放。

## 一、秒殺系統的核心問題

### 1.1 什麼是秒殺

- 秒殺本質上是「在很短的時間內，很多人同時搶少量資源」。
- 典型場景像搶票、搶限量商品、搶活動名額。
- 難點不在單純的請求數量，而在請求會集中爆發。

### 1.2 主要風險

- 流量暴增會直接把服務打垮。
- 庫存扣減如果處理不好，會出現超賣。
- 如果所有請求都直接打資料庫，DB 很容易成為瓶頸。
- 使用者在高峰期等待太久，體驗會非常差。

### 1.3 設計目標

- 系統要能扛住大流量。
- 庫存要準確，不能超賣，也不能少賣。
- 下單流程要盡量快，最好先回應，再異步處理後續流程。
- 失敗請求要能快速被攔截，不要一路打到核心資料層。

## 二、大型高並發架構

### 2.1 分布式集群

- 秒殺系統通常不是單機服務，而是分布式集群。
- 透過多台服務器分擔壓力，單點故障風險也比較低。
- 高可用通常會搭配容災、備援與節點容錯。

### 2.2 多層負載均衡

- 文章提到流量會經過多層負載均衡。
- 常見組合可以理解成前面先有網路層分流，再到應用層分流。
- 這樣可以把請求分散到不同機器，避免單點過載。

### 2.3 流量削峰

- 秒殺流量不是平均的，而是瞬間集中。
- 架構設計要做的事，是把「同時到達」的流量變得可控。
- 常見做法是排隊、限流、預熱、預扣與異步化。

## 三、庫存扣減思路

### 3.1 不能直接打資料庫

- 如果每個請求都直接查庫存、扣庫存，DB 會先被打爆。
- 高並發秒殺最怕核心資料層直接暴露在流量洪峰下。
- 所以要先在更前面的層級做攔截和緩衝。

### 3.2 預扣庫存

- 預扣庫存是這類系統最核心的思路之一。
- 請求進來時先把庫存扣掉，先保證不超賣。
- 後續再異步生成訂單，減少請求同步等待時間。
- 如果最後訂單失敗或超時，則把庫存釋放回去。

### 3.3 本地扣庫存

- 為了進一步減少對中心庫存服務的壓力，可以先把熱庫存同步到本地。
- 本地先做快速判斷，能通過再往後走。
- 這種方式能大幅降低中心系統的壓力。
- 但也要考慮各節點之間的一致性問題。

### 3.4 Redis 扣庫存

- Redis 常被用來承接高頻扣庫存請求。
- 透過原子操作先完成扣減，可以避免多請求同時改同一份庫存資料。
- Redis 的定位通常是前置緩衝，不是最終的單一真相來源。
- 最終落庫時仍需要和訂單、支付狀態一起做一致性處理。

## 四、訂單流程

### 4.1 先回應，再處理

- 秒殺場景最怕使用者一直卡在同步等待。
- 一般會先回應「已受理」或「排隊中」。
- 真正的訂單生成放到後台異步處理。

### 4.2 異步下單

- 下單流程常會透過 MQ 這類消息隊列來解耦。
- 前端請求進來後先完成搶購判斷，再把訂單消息送去後台。
- 這樣能把瞬間流量轉成可控的持續處理。

### 4.3 訂單超時釋放

- 如果使用者拿到名額但沒有在有效時間內付款，訂單就會失效。
- 失效後要把庫存釋放回來，避免資源被白白占住。
- 這也是防止少賣和資源浪費的重要機制。

## 五、常見保護機制

### 5.1 限流

- 限流是秒殺系統的基本保護。
- 可以限制單 IP、單帳號、單介面或單節點流量。
- 目的不是把所有人擋掉，而是避免核心服務被打穿。

### 5.2 排隊

- 排隊是把「同時處理」改成「有序處理」。
- 對秒殺來說，排隊通常比硬扛更穩。
- 使用者體感可能慢一點，但系統整體更安全。

### 5.3 快取與預熱

- 熱門商品資訊通常會提前預熱到快取。
- 這樣請求進來時，不需要頻繁打 DB。
- 對活動開始前的準備特別重要。

### 5.4 防重複提交

- 同一個使用者不應該在短時間內重複提交多次。
- 可以透過 token、請求 id、狀態位或其他機制防重放。
- 這不只是防刷，也是在保護業務流程正確性。

## 六、這篇常見結論

- 秒殺系統的核心不是單點優化，而是整體架構設計。
- 最重要的事情通常是：削峰、限流、預扣庫存、異步下單、超時釋放。
- 直接打資料庫是最危險的設計，因為高峰流量會把它先打垮。
- 真正穩定的秒殺架構，目標是讓大量請求先被前置系統吸收，再慢慢往後處理。
