# [FE] System - 1 大型項目前端架構

## 這篇在講什麼

- 這篇文章不是在講單一技術，而是在講大型項目要怎麼從系統層面設計前端架構。
- 核心精神是：架構不是為了炫技，而是為了降低風險、提高可維護性、提高交付效率。
- 內容主要分成兩塊：
- 基礎層：偏基礎設施與工程治理。
- 應用層：偏產品頁面、團隊協作與實際業務承載。

## 一、綜合

### 1.1 使用場景

- 適合單個或多個大型項目。
- 適合前端人力超過 10 人以上的團隊。
- 團隊越大、歷史包袱越重、需求越複雜，架構收益通常越明顯。
- 如果是小團隊或短期專案，有些設計不一定划算，不能硬套。

### 1.2 核心思想

- 解決問題：架構設計是拿來解決已存在或未來可能發生的技術問題。
- 人效比：要衡量額外開發成本，和它未來能省下多少時間、金錢、風險。
- 定性與定量：架構改善最好能量化；就算不能量化，也要能清楚說明收益。
- 數據敏感：推動架構設計時，最好用埋點、日誌、報表等數據佐證。

### 1.3 切入角度

- 分成基礎層與應用層。
- 基礎層偏基礎設施，與業務耦合較低。
- 應用層更接近使用者，主要解決實際產品問題。
- 有些設計同時沾到兩邊，實務上可以依主要價值歸類。

### 1.4 其他

- 這篇內容其實不只屬於前端，還牽涉後端、運維、測試與資料。
- 負責架構的人要有比較完整的技術視野，不能只看前端頁面本身。
- 文章的描述方式很有代表性：先講項目，再講解決的問題，再講實際意義。

## 二、基礎層設計

### 2.1 自建 Gitlab

- 使用自建 Gitlab 來做版本與權限管理，比把代碼散落在不同平台更好控。
- 好處包含提交紀錄、權限管理、流程管理、以及後續插件整合。
- 對大型團隊來說，代碼是重要資產，平台掌握在自己手上更安全。

### 2.2 版本管理

- 核心是建立可追蹤、可回滾、可鎖版本的流程。
- 發版後分支應盡量鎖死，避免版本被偷偷改掉。
- 版本要支援多版並存，避免一出問題就整個回不去。
- 全自動發布流程比手動上版更穩定，也更省人力。

### 2.3 自動編譯發布 Jenkins

- 文章把 Jenkins 視為自動化發布流程的一環。
- 目標不是「有 CI/CD」而已，而是讓開發者不直接接觸部署環境。
- 自動化發布可以降低人為失誤，也能加快交付速度。
- 對大型團隊來說，這是把上線流程標準化的重要手段。

### 2.4 純前端版本發布

- 重點是前端靜態資源也要有版本治理，而不是只靠手動改路徑。
- 對大型站點來說，前端版本發佈、回滾、灰度都應該是可控流程。
- 這樣當某版資源有問題時，回切成本才會低。

### 2.5 統一腳手架

- 適合有很多獨立中小項目的團隊。
- 可以降低每次開新專案都要重配環境的成本。
- 統一目錄結構與建置方式，讓維護、交接、培訓更容易。
- 也能幫團隊建立比較一致的技術基線。

### 2.6 Node 中間層

- 適合需要 SEO，或前端要更深度參與後端資料處理的情境。
- 可以讓前端接觸更多業務邏輯，提升開發效率與彈性。
- 但風險也更高，做不好可能導致更大範圍故障。
- 對大型業務來說，這是一把雙面刃。

### 2.7 埋點系統

- 強烈建議前端建立自己的埋點系統。
- 可用來看 PV、UV、功能使用量、錯誤情況。
- 這不只是統計數據，而是前端理解業務與使用者行為的基礎。
- 有了數據，後續優化與決策才會比較站得住腳。

### 2.8 監控和報警系統

- 監控不只是後端的事，前端也應該能看自己的健康狀況。
- 可以監控報錯量、訪問量變化、特定功能的使用狀況。
- 異常時能及早發現，避免用戶先撞到問題才被動處理。
- 對大型產品來說，這是穩定性的重要保底。

### 2.9 安全管理

- 前端安全通常要跟後端一起做，不可能只靠前端獨立解決。
- 常見包括 XSS、HTTPS、CSRF 等問題。
- 常見原則是：不要相信使用者輸入，不要亂用 `eval`，敏感流程要有伺服器端防護。
- 架構層上要把風險前置，而不是等被打才修。

### 2.10 Eslint

- 可以降低低級 bug。
- 可以強化可維護性和可讀性。
- 也能統一團隊寫法，讓多人協作更穩。
- 在大型團隊裡，這不是小事，而是基礎治理。

### 2.11 灰度發布

- 先讓小部分流量使用新版本，確認沒有問題再擴大。
- 適合主鏈路、流量大、風險高的頁面。
- 可以控制風險，也能用埋點觀察新版本的效果。
- 灰度本質上是在生產環境做小範圍試驗。

### 2.12 前後端分離

- 這裡不是指一般人常說的 API 分離，而是職責邊界的分離。
- 大型項目中，HTML 可以交給後端處理，前端負責靜態資源與頁面邏輯。
- 好處是可以統一插入埋點、監控、國際化、頁面標識等內容。
- 也更利於灰度與版本控制。

### 2.13 Mock

- 當後端接口還沒好時，Mock 能讓前後端並行開發。
- 文中偏向把 Mock 放在可以快速調整的方案裡。
- 重點不是一定要建一個龐大平台，而是讓對接成本最低。
- 在協作效率上很有價值。

### 2.14 定期備份

- 很容易被忽略，但一旦出事損失很大。
- 包含機器損壞、誤刪、資料庫事故、致命 bug。
- 常見做法是定期備份、多機備份、容災建設。
- 這屬於平常看不到、出事時很值錢的能力。

## 三、應用層設計

### 3.1 多頁和單頁

- 大型項目通常更偏向多頁架構。
- 原因是頁面之間獨立，重構、回滾、版本控制更容易。
- 單頁架構在大型歷史項目中，常因為耦合太高而變得難維護。
- 多頁的缺點是切換時可能會有白屏，但通常可接受。

### 3.2 以應用為單位劃分前端項目

- 大型團隊不適合把所有東西塞在同一個大倉庫裡讓每個人亂改。
- 以應用為單位切分，有助於權限管理、發布管理與責任界定。
- 每個應用最好都有自己的邊界，不要讓一個需求牽一堆無關頁面。
- 這樣比較利於長期維護。

### 3.3 基礎元件庫建設

- 團隊大、需求多時，做一套自己的元件庫通常值得。
- 但前提是團隊規模與技術成熟度要夠，不然維護成本會反噬。
- 元件庫要有文檔、版本控制、可擴充性，最好也要有設計協作。
- 這類建設的目的，是統一風格與重複利用能力。

### 3.4 技術棧統一

- 團隊如果同時混很多框架、很多 UI 套件，最後會變成大雜燴。
- 因此技術選型要盡量統一。
- 這包含框架、UI 套件、特殊第三方庫版本、工程基礎設施。
- 好處是招人、培訓、交接、維護都更容易。

### 3.5 瀏覽器兼容

- 大型項目常會遇到不同瀏覽器的怪問題。
- 可透過 postcss、webpack loader、統一寫法、測試策略來減少問題。
- 最終還是要根據目標使用者環境決定支援範圍。
- 兼容性不是附加題，而是產品邊界的一部分。

### 3.6 內容平台建設

- 文章提到應建立內部論壇或博客站點。
- 這能幫助團隊沉澱知識、分享經驗、保留歷史。
- 也能降低資訊分散帶來的溝通成本。
- 對大型公司來說，這是內部技術文化的一部分。

### 3.7 權限管理平台

- 當公司人多時，權限管理不能只靠人手分配。
- 要有自動化申請、審批、失效、離職回收等流程。
- 權限最好有時效性，不要無限期保留。
- 這種平台主要是在讓公司流程正規化與可視化。

### 3.8 登錄系統設計（單點登錄）

- 當業務線多、系統多時，SSO 能減少重複登入與權限割裂。
- 使用者體驗更好，管理也更方便。
- 對多系統共用帳號體系的場景很重要。
- 也能降低每個系統都重做一套登入模組的成本。

### 3.9 CDN

- CDN 可以提升靜態資源載入速度。
- 對跨地區、跨運營商的使用者特別有幫助。
- 也能降低源站壓力，並幫助抵禦部分攻擊與流量波動。
- 大型站點幾乎都離不開。

### 3.10 負載均衡

- 透過負載均衡，可以分散單台 server 壓力。
- 對大流量、活動頁、業務高峰場景尤其必要。
- 這會直接影響可用性、擴展性與穩定性。

### 3.11 多端共用一套接口

- 同一個業務若有 PC 與 H5，盡量共用同一套接口。
- 這樣可以減少重複開發與接口分裂。
- 若真有特殊流程，才獨立出專用接口。
- 這是降低維護成本和重複工作的典型做法。

## 四、總結

- 這篇文章的重點不是某個工具，而是大型前端團隊如何從系統設計角度做治理。
- 內容的主軸是：減少風險、提高效率、建立可擴展與可維護的工程體系。
- 實際上不是每個公司都要一次做滿，但至少要知道這些能力各自解決什麼問題。
- 架構設計的核心，不是做更多，而是做對的事情。
